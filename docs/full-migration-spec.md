# Танцуй: полная спецификация для переноса проекта

## 1. Назначение документа

Этот документ описывает текущий MVP приложения `Танцуй`, его пользовательские сценарии, структуру данных, видеопайплайн, анализ позы, систему оценки и рекомендуемую архитектуру нового проекта.

Документ нужно использовать как техническое задание при переносе приложения:

- в React, Next.js, Vue, Flutter или другой клиент;
- в мобильное приложение через Capacitor;
- на новый backend;
- в другой репозиторий;
- из локального прототипа в полноценный многопользовательский продукт.

Важно различать:

- `Реализовано` — работает в текущем коде;
- `Демо` — интерфейс существует, но использует статические данные;
- `Запланировано` — зафиксированная продуктовая логика, которую нужно реализовать на backend или в следующих версиях.

## 2. Идея продукта

`Танцуй` — платформа для танцевальных студий, педагогов и учеников.

Основной сценарий:

1. Педагог создает студию.
2. В студии создаются группы и вложенные подгруппы.
3. В группе или подгруппе педагог создает урок.
4. Для урока загружаются два видео:
   - обучающее видео с объяснением;
   - эталонное видео с танцем под музыку.
5. Педагог задает начало и конец полезного фрагмента.
6. Из видео под музыку извлекается эталонный скелет.
7. Ученик изучает объяснение и танец.
8. Ученик записывает или загружает собственное видео.
9. Из видео ученика извлекается скелет.
10. Скелет ученика сравнивается с эталоном.
11. Ученик получает процент, звезды, метрики и грубые ошибки.
12. Педагог видит результаты учеников.

## 3. Текущий технологический стек

Текущая версия не использует сборщик или framework.

- HTML: `index.html`
- CSS: `styles.css`
- JavaScript ES modules: `app.js`
- PWA manifest: `manifest.webmanifest`
- Service Worker: `sw.js`
- Pose estimation: MediaPipe Tasks Vision `0.10.35`
- Pose model: `pose_landmarker_full`, float16
- Хранение MVP: `localStorage`
- Видео: браузерные `HTMLVideoElement`, `Blob URL`
- Наложение скелета: Canvas 2D
- Камера: `navigator.mediaDevices.getUserMedia`
- Деплой: GitHub + Netlify

```text
https://cdn.jsdelivr.net/npm/@mediapipe/tasks-vision@0.10.35
https://unpkg.com/@mediapipe/tasks-vision@0.10.35
```

Модель:

```text
https://storage.googleapis.com/mediapipe-models/pose_landmarker/pose_landmarker_full/float16/latest/pose_landmarker_full.task
```

## 5. Роли

### 5.1. Ученик

Ученик:

- создает профиль;
- автоматически вступает в студию по приглашению;
- видит свою студию, имя, уровень и звезды;
- выбирает урок;
- смотрит обучающее видео;
- смотрит эталон под музыку;
- включает фронтальную камеру или загружает видео;
- отмечает начало и конец;
- извлекает скелет;
- сдает танец;
- получает оценку;
- смотрит историю уровней и попыток.

### 5.2. Педагог

Педагог:

- создает одну основную студию;
- оформляет профиль студии;
- создает группы;
- создает вложенные подгруппы;
- создает уроки;
- загружает два видео урока;
- размечает полезные фрагменты;
- создает и проверяет эталон;
- видит список учеников и статусы прохождения.

### 5.3. Владелец или администратор студии

В текущем MVP отдельная роль не реализована.

В полной версии владелец:

- управляет студией;
- добавляет педагогов;
- назначает права;
- управляет группами;
- видит аналитику студии;
- управляет подпиской и оплатой.

## 6. Кабинет педагога

### 6.1. Студия как постоянное пространство

После создания студия становится постоянным верхним пространством. Выйти в список студий нельзя. У владельца MVP может быть только одна студия.

Шапка студии постоянно видна во всех внутренних разделах:

- обложка;
- круглый логотип;
- название;
- описание;
- адрес;
- направления;
- владелец;
- команда педагогов;
- внешняя ссылка.

Группы, подгруппы, уроки и рабочие инструменты всегда располагаются ниже шапки.

### 6.2. Создание студии

Поля:

- название;
- описание;
- логотип;
- широкая обложка;
- адрес;
- направления;
- сайт или социальная сеть.

Обложка:

- уменьшается максимум до `1600 × 1000`;
- сохраняется как JPEG с качеством `0.84`;
- показывается через `object-fit: cover`;
- всегда заполняет шапку;
- лишние части обрезаются по центру.

### 6.3. Настройки студии

Настройки позволяют изменить:

- логотип;
- обложку;
- название;
- описание;
- направления;
- адрес;
- контакт или ссылку.

Форма прокручивается внутри модального окна. Нижние кнопки закреплены.

### 6.4. Группы и подгруппы

Иерархия:

```text
Студия
└── Группа
    ├── Урок
    └── Подгруппа, уровень 1
        ├── Урок
        └── Подгруппа, уровень 2
            ├── Урок
            └── Подгруппа, уровень 3
                └── Урок
```

Максимальная глубина подгрупп: три уровня внутри основной группы.

Карточка группы или подгруппы показывает:

- крупную круглую иконку;
- номер, если цифра найдена в названии;
- тип: `Группа`, `Подгруппа / партия`, `Урок`;
- крупное название;
- описание, возраст или уровень;
- стрелку входа.

### 6.5. Навигация

Используется древовидная логика:

- активная студия;
- активная группа;
- массив активных подгрупп;
- активный урок.

В шапке раздела есть:

- breadcrumb;
- название текущего пространства;
- кнопка назад;
- кнопка редактирования;
- кнопка создания подгруппы;
- кнопка создания урока или группы.

Шапка студии при переходах не исчезает.

## 7. Подготовка урока педагогом

### 7.1. Два независимых видео

Урок содержит:

1. `learningVideo` — обучающее объяснение.
2. `examVideo` — готовый танец под музыку.

Скелет строится только из `examVideo`.

### 7.2. Маркеры

У каждого видео свой диапазон:

```json
{
  "duration": 0,
  "start": 0,
  "end": 0,
  "startSet": false,
  "endSet": false
}
```

Маркеры нужны, чтобы убрать:

- включение камеры;
- отход педагога на позицию;
- включение музыки;
- подход к камере после танца;
- лишние слова до или после объяснения.

### 7.3. Пайплайн подготовки

Этапы:

1. Видео.
2. Эталон.
3. Проверка.
4. Фрагмент.
5. Публикация.

Кнопка публикации доступна, когда:

- оба видео загружены;
- диапазоны размечены;
- эталон построен.

Фактическое сохранение опубликованного урока на сервер сейчас не реализовано.

## 8. MediaPipe и построение скелета

### 8.1. Два экземпляра модели

Создаются:

- `teacherLandmarker` с режимом `IMAGE`;
- `liveLandmarker` с режимом `VIDEO`.

Настройки:

```json
{
  "numPoses": 1,
  "minPoseDetectionConfidence": 0.45,
  "minPosePresenceConfidence": 0.45,
  "minTrackingConfidence": 0.45
}
```

Сначала используется GPU. При ошибке выполняется fallback на CPU.

### 8.2. Эталон

Эталон сканируется с частотой 30 кадров в секунду.

Для каждого найденного кадра сохраняются:

```json
{
  "time": 1.2,
  "displayLandmarks": [],
  "landmarks": [],
  "angles": {},
  "signature": {}
}
```

- `displayLandmarks` — исходные точки MediaPipe для отрисовки;
- `landmarks` — нормализованные точки;
- `angles` — углы суставов;
- `signature` — компактное описание формы и амплитуды.

### 8.3. Нормализация тела

Чтобы уменьшить влияние роста, комплекции, расстояния до камеры и положения в кадре:

- центр задается серединой бедер;
- все точки смещаются относительно центра;
- масштаб задается расстоянием между левым плечом и правым бедром;
- координаты `x`, `y`, `z` делятся на масштаб.

Это не делает систему полностью независимой от ракурса камеры, но убирает большую часть различий размера и позиции.

### 8.4. Синхронное наложение эталона

При воспроизведении используется `requestVideoFrameCallback`.

Между двумя ближайшими кадрами эталона выполняется линейная интерполяция:

```text
displayPose(t) = lerp(previousPose, nextPose, progress)
```

Это уменьшает визуальное отставание скелета и делает наложение плавнее.

### 8.5. Защита от неверных линий скелета

При отрисовке:

- скрываются мелкие точки лица;
- скрываются дополнительные точки кистей;
- проверяется видимость точки;
- проверяется длина сегмента в пикселях;
- не рисуются нелогично длинные связи;
- не рисуются точки, схлопнувшиеся в корпус;
- цвет отдельных костей может различаться.

## 9. Видео ученика

### 9.1. Источники

Ученик может:

- включить фронтальную камеру;
- загрузить файл `video/*`.

Для фронтальной камеры запрашивается:

```json
{
  "facingMode": "user",
  "width": 1280,
  "height": 720,
  "audio": false
}
```

Если фронтальная камера недоступна, запрашивается любая камера.

Камера требует HTTPS или localhost.

### 9.2. iPhone и Safari

Для видео используются:

- `playsinline`;
- `webkit-playsinline`;
- `muted`;
- `preload="auto"`;
- принудительный показ первого кадра после загрузки.

### 9.3. Скелет ученика

После выбора диапазона ученик нажимает `Считать скелет`.

Сохраняются:

```json
{
  "time": 1.2,
  "displayLandmarks": [],
  "landmarks": [],
  "angles": {},
  "signature": {}
}
```

После считывания скелет можно воспроизводить поверх видео без повторного анализа каждого кадра.

## 10. Сценарий обучения и сдачи

### 10.1. Обучение

- используется обучающее видео;
- скелет педагога отключен;
- видео можно замедлять от `0.4x` до `1x`;
- можно начинать с выбранного фрагмента;
- основной экран фокусируется на крупном видео педагога.

### 10.2. Сдача

1. Ученик нажимает `Сдать`.
2. Показывается TikTok-подобный отсчет.
3. Ученик успевает занять исходную позу.
4. Запускается видео педагога под музыку.
5. Одновременно записываются позы ученика.
6. Запись завершается в конце выбранного диапазона.
7. Выполняется сравнение.
8. Показывается результат.

## 11. Алгоритм сравнения

### 11.1. Принцип

Система не сравнивает абсолютные экранные координаты. Основные признаки:

- категории положения рук;
- сгиб рук;
- сгиб ног;
- ширина стойки;
- наклон и уровень корпуса;
- форма позы;
- амплитуда;
- направление движения;
- скорость и энергия;
- тайминг.

### 11.2. Привязка по времени

Для каждого кадра эталона ищется ближайший кадр ученика.

Допустимое расстояние:

```text
0.28 секунды
```

Если кадр не найден, момент получает нулевую оценку.

### 11.3. Подпись движения

`movementSignature` содержит:

```json
{
  "leftArm": "straight_up_open",
  "rightArm": "soft_middle_near",
  "leftLeg": "straight",
  "rightLeg": "bent",
  "stance": 1.2,
  "torso": "upright",
  "level": -0.8,
  "armSpread": 2.1,
  "reach": 1.0
}
```

### 11.4. Типы момента

Каждый момент классифицируется как:

- `accent`;
- `transition`;
- `hold`.

Для разных моментов используются разные веса.

Акцент:

```json
{
  "position": 0.54,
  "timing": 0.24,
  "trajectory": 0.04,
  "amplitude": 0.12,
  "energy": 0.06
}
```

Переход:

```json
{
  "position": 0.44,
  "timing": 0.20,
  "trajectory": 0.08,
  "amplitude": 0.18,
  "energy": 0.10
}
```

Удержание:

```json
{
  "position": 0.58,
  "timing": 0.22,
  "trajectory": 0.02,
  "amplitude": 0.14,
  "energy": 0.04
}
```

### 11.5. Защита от дрожания точек

Если основная форма тела похожа:

- траектория поднимается минимум до 92;
- энергия поднимается минимум до 90.

При средней похожести:

- траектория поднимается минимум до 84;
- энергия поднимается минимум до 80.

Так мелкое дрожание MediaPipe не становится ошибкой ученика.

### 11.6. Калибровка результата

Система намеренно мягкая.

При хорошем покрытии и высоком качестве результат повышается до 100%.

Критерий уровня эталона:

- coverage не ниже 90%;
- core pose не ниже 82;
- timing не ниже 82;
- итог не ниже 82.

Смысл: идентичное или практически идентичное видео должно получать 100%.

### 11.7. Ошибки

Показываются только грубые ошибки:

- score ниже 50;
- соседние ошибки ближе 0.8 секунды объединяются;
- показывается не более шести ошибок;
- при результате от 92 и coverage от 96% ошибки скрываются.

Возможные подписи:

- руки отличаются от эталона;
- ноги отличаются от эталона;
- корпус не совпал;
- не попал в музыку;
- траектория движения другая;
- амплитуда отличается;
- скорость или энергия отличается;
- форма позы отличается.

## 12. Результат, звезды и уровни

### 12.1. Звезды за попытку

```text
90-100% = 5 звезд
78-89%  = 4 звезды
62-77%  = 3 звезды
42-61%  = 2 звезды
20-41%  = 1 звезда
0-19%   = 0 звезд
```

### 12.2. Результат

Показываются:

- процент совпадения;
- звезды;
- покрытие кадров;
- общий текст результата;
- текущий уровень;
- рекомендации;
- кнопка ошибок, только если ошибки есть.

### 12.3. Метрики

В интерфейсе:

- руки;
- ноги;
- корпус;
- грудь;
- амплитуда;
- энергия;
- траектория;
- ритм.

Примечание: текущие `head` и `fingers` в расчетах являются заглушками со значением 70. В UI эти позиции переиспользованы под амплитуду и энергию.

### 12.4. Уровни

Реализовано 30 уровней. Уровень вычисляется по суммарному количеству звезд всех попыток.

Диапазон:

- уровень 1: 0 звезд;
- уровень 10: 110 звезд;
- уровень 20: 420 звезд;
- уровень 30: 1100 звезд.

Полный массив `LEVELS` находится в `app.js`.

## 14. Список учеников урока

Текущий UI показывает:

- имя и фамилию;
- никнейм;
- статус;
- процент;
- звезды.

Статусы:

- не начал;
- посмотрел;
- учит;

## 21. QR-приглашения

Запланированный сценарий:

1. Педагог создает приглашение для группы.
2. Backend генерирует случайный токен.
3. В QR помещается URL с токеном.
4. Ученик открывает ссылку.
5. Если аккаунта нет, регистрируется.
6. После регистрации автоматически вступает в группу.

Таблица:

```text
id

group_id
teacher_id

max_uses
used_count
created_at
```

В QR нельзя помещать персональные данные или открытые ID без серверной проверки.

## 24. Ограничения текущего MVP

- один большой `app.js`;
- нет TypeScript;
- нет unit и integration tests;
- нет backend;
- нет авторизации;
- MediaPipe требует интернет при первой загрузке;
- localStorage ограничен по объему;
- Data URL изображений быстро расходуют localStorage;
- Blob URL видео живут только в текущей вкладке;
- монокулярная 2D/псевдо-3D поза зависит от ракурса;
- нет анализа музыкальных акцентов;
- камера должна быть неподвижна;
- в кадре желательно видеть человека полностью;
- сложные перекрытия конечностей снижают точность;
- Safari может требовать пользовательское действие для воспроизведения.

## 26. Обязательные acceptance tests

### Студия

- студия создается только после заполнения названия;
- логотип виден целиком;
- обложка заполняет ширину;
- шапка не исчезает в группе, подгруппе и уроке;
- настройки сохраняются после перезагрузки.

### Дерево

- можно создать несколько групп;
- можно войти в группу;
- можно вернуться в студию;
- можно создать подгруппы до третьего уровня;
- четвертый уровень запрещен;
- урок можно создать на любом разрешенном уровне.

### Видео педагога

- оба видео открываются;
- диапазоны независимы;
- обучающее видео не анализируется;
- эталон строится только из exam;
- эталон сохраняется и повторно загружается;
- скелет синхронен с видео.

### Видео ученика

- видео открывается на iPhone;
- доступна фронтальная камера;
- диапазон работает;
- скелет строится;
- скелет воспроизводится синхронно;
- сдача завершается в конце диапазона.

### Оценка

- одинаковое видео дает 100%;
- небольшое дрожание не создает ошибку;
- разный масштаб тела не снижает результат критически;
- пропущенные кадры снижают coverage;
- показываются только грубые ошибки;
- кнопка ошибок скрыта, если ошибок нет.

## 27. Критические правила, которые нельзя потерять

1. Обучающее и эталонное видео — разные сущности.
2. Скелет эталона строится только из видео под музыку.
3. У обоих видео свои маркеры.
4. Эталон строится один раз и сохраняется.
5. Скелет накладывается синхронно по времени видео.
6. Сравнение не должно зависеть от абсолютного положения и роста.
7. Оценка не должна наказывать за микродрожание модели.
8. Идентичное видео должно давать 100%.
9. Ошибки показываются только при грубом расхождении.
10. Студия является постоянной оболочкой кабинета педагога.
11. Все группы, подгруппы и уроки находятся ниже шапки студии.
12. Вложенность подгрупп ограничена тремя уровнями.
13. Камера ученика должна работать через HTTPS на iPhone.
14. Пользовательские видео и персональные данные нельзя хранить только во frontend.

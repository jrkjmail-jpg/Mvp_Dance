# Танцуй — глобальный план достройки MVP

Документ фиксирует текущее понимание проекта и направление миграции от визуального frontend-MVP к масштабируемой платформе для танцевальных студий.

Дата фиксации: 2026-07-10.

## 1. Текущее состояние

Проект сейчас представляет собой статический MVP:

- основной интерфейс находится в `index.html`;
- основная логика приложения находится в `app.js`;
- большая часть визуального слоя находится в `styles.css`;
- service worker находится в `sw.js`;
- данные временно хранятся в `localStorage`;
- деплой идет через GitHub и Cloudflare Pages;
- MediaPipe работает на стороне клиента.

MVP уже достаточно сильный визуально, но архитектурно он пока больше похож на прототип, чем на платформу. Главный риск: слишком много логики, UI, состояния, MediaPipe и бизнес-сценариев смешано в одном большом frontend-коде.

## 2. Главная цель следующего этапа

Не продолжать хаотично полировать отдельные экраны, а сначала зафиксировать структуру продукта:

- роли;
- сущности;
- сценарии;
- модель данных;
- права доступа;
- хранение данных;
- юридически чувствительные места;
- будущий backend;
- план миграции текущего MVP.

После этого уже безопаснее продолжать дизайн и функциональные доработки.

## 3. Основные роли

### Платформа

Владелец цифрового продукта. Предоставляет студиям электронное пространство, инфраструктуру, интерфейс, хранение, аналитику и доступ к функционалу.

### Студия

Клиент платформы в B2B-модели. Студия ведет учеников, создает группы, уроки, приглашения, управляет доступом и взаимодействует с родителями/учениками.

### Педагог

Пользователь со стороны студии. Создает уроки, загружает видео обучения и эталона, проверяет сдачи, видит учеников и прогресс.

### Ученик

Пользователь, который смотрит уроки, тренируется, сдает попытки, получает оценку и прогресс.

### Родитель / законный представитель

Нужен на этапе регистрации несовершеннолетнего ученика. Может подтверждать согласия, заполнять анкету, после чего ученик получает доступ к своему кабинету.

На первом этапе отдельный полноценный родительский кабинет можно не делать, но юридический шаг подтверждения родителем нужно заложить.

## 5. Ключевые сценарии

### 5.1. Приглашение ученика

1. Студия создает приглашение.
2. Указывает имя, фамилию, группу или предполагаемую группу.
3. Система генерирует ссылку/QR-код.
4. Родитель открывает ссылку.
5. Родитель заполняет анкету и подтверждает согласия.
6. Ученик получает доступ к кабинету.
7. Студия видит статус ученика: ожидает регистрации / активен / отключен.

### 5.2. Создание урока

1. Педагог создает урок.
2. Загружает видео обучения.
3. При необходимости выставляет начало и конец.
4. Загружает видео-эталон.
5. При необходимости выставляет начало и конец.
6. Система считывает эталон.
7. Педагог проверяет результат.
8. Урок публикуется в нужную группу или подгруппу.

### 5.3. Прохождение урока учеником

1. Ученик открывает доступную группу.
2. Видит только те уроки, к которым у него есть доступ.
3. Открывает урок.
4. Смотрит обучение.
5. Переходит к сдаче.
6. Сдает попытку через камеру или загруженное видео.
7. Получает оценку, звезды и расшифровку.
8. Результат сохраняется.

## 6. Данные и хранение

Сейчас данные находятся в `localStorage`, но для реальной платформы их нужно перенести на backend.

`localStorage` можно оставить только для:

- темы интерфейса;
- временных настроек;
- локального кэша;
- черновиков.

На сервере должны храниться:

- студии;
- ученики;
- группы;
- уроки;
- статусы доступа;
- приглашения;
- согласия;
- результаты;
- попытки;
- метаданные видео;
- точки скелета;
- журналы действий.

## 7. Видео и детские данные

Проект работает с чувствительными данными, особенно если ученики несовершеннолетние.

Желательный подход:

- по возможности анализировать видео на устройстве;
- не хранить видео ученика по умолчанию;
- хранить результат, точки скелета и метаданные попытки;
- хранить само видео только при отдельном согласии и реальной необходимости;
- предусмотреть удаление данных;
- предусмотреть срок хранения;
- хранить данные в России;
- вести журнал согласий и действий.

Юридическую модель нужно отдельно согласовать со специалистом по персональным данным и работе с несовершеннолетними.

## 8. Backend и инфраструктура

Для следующего этапа нужна инфраструктура в России:

- VPS/VDS или выделенный сервер;
- backend API;
- PostgreSQL;
- S3-совместимое объектное хранилище для видео и изображений;
- хранение skeleton-данных отдельно от видео;
- авторизация;
- роли и права доступа;
- резервные копии;
- мониторинг;
- журнал действий.

Cloudflare Pages можно временно оставить для frontend, но если требуется строгое хранение и обработка данных в России, нужно заранее спроектировать российскую инфраструктуру и понять, какие части можно оставлять на Cloudflare, а какие нельзя.

## 9. Безопасность

Минимально необходимые элементы:

- HTTPS;
- серверная авторизация;
- роли: платформа, студия, педагог, ученик, родитель;
- серверная проверка доступа к урокам;
- защищенные invite-ссылки;
- срок жизни приглашений;
- возможность отозвать приглашение;
- аудит действий;
- защита от перебора;
- резервные копии;
- удаление данных по запросу;
- разграничение доступа между студиями.

Важно: клиентский код не должен быть источником истины для прав доступа. Все права должны проверяться на сервере.

## 10. Архитектурный долг текущего MVP

Главные слабые места текущей кодовой базы:

- один большой `app.js`;
- один большой `styles.css`;
- смешение UI, состояния, бизнес-логики и MediaPipe;
- много точечных CSS-переопределений;
- сложность безопасно менять мобильную версию;
- нет автоматических тестов;
- нет backend;
- нет настоящей модели пользователей;
- нет серверных прав доступа;
- нет нормальной документации сценариев.

## 12. Порядок дальнейшей работы

### Этап 1. Стабилизация MVP

- зафиксировать текущий UI как baseline;
- убрать случайные временные файлы или добавить их осознанно;
- восстановить документацию;
- перестать делать большие хаотичные правки в `app.js` и `styles.css`.

### Этап 2. Описание продукта

- финализировать роли;
- описать сценарии;
- описать модель данных;
- описать права доступа;
- описать юридические точки согласий.

### Этап 3. Backend

- выбрать российскую инфраструктуру;
- поднять API;
- подключить базу;
- подключить хранилище;
- реализовать авторизацию;
- реализовать студии, группы, уроки, учеников.

### Этап 4. Перенос данных

- заменить `localStorage` на API;
- оставить локально только настройки и кэш;
- сделать миграционный слой, чтобы интерфейс не развалился.

### Этап 5. Рефакторинг frontend

- выделить компоненты;
- выделить MediaPipe;
- выделить scoring;
- выделить кабинеты;
- выделить дизайн-систему.

### Этап 6. Тестирование

- добавить сценарное тестирование;
- проверить iPhone Safari;
- проверить iPad Safari;
- проверить Android Chrome;
- проверить desktop Chrome/Safari;
- добавить smoke-тесты основных сценариев.

## 13. Что делать в первую очередь

Самый правильный следующий шаг:

1. Зафиксировать стабильную версию MVP.
2. Привести документацию в порядок.
3. Описать модель данных.
4. Спроектировать backend и хранение в России.
5. После этого уже аккуратно переносить функционал на сервер.

Главное правило: дальше не строить платформу только через правки интерфейса. Сначала нужна структура, потом реализация.

# Refactor checklist

Чеклист нужен, чтобы аккуратно переносить код из монолитного MVP и не ломать рабочий интерфейс.

## 1. Перед любым переносом

- Проверить `git status`.
- Понять, какие файлы будут затронуты.
- Не трогать несвязанные файлы.
- Сформулировать одну цель изменения.
- Не смешивать рефакторинг и новую функцию.

## 2. После любого переноса

Проверить вручную:

- приложение открылось;
- ошибок в консоли нет;
- шапка отображается корректно;
- переключение кабинета работает;
- кабинет ученика открывается;
- кабинет педагога открывается;
- страница студии открывается;
- урок открывается;
- мобильная версия не развалилась.

## 3. Первый безопасный слой для выноса

Начинать лучше с `storage`, потому что он меньше влияет на визуал.

Что можно вынести:

- чтение темы;
- сохранение темы;
- сохранение профиля ученика;
- сохранение данных студии;
- сохранение попыток;
- сохранение избранных уроков.

## 4. Второй слой

После storage можно выносить `scoring`.

Что можно вынести:

- расчет звезд;
- расчет процента;
- текст расшифровки;
- утилиты оценки.

Важно: сначала только чистые функции, без DOM.

## 5. Третий слой

После scoring можно трогать `pose`.

Этот слой опаснее, потому что влияет на скорость видео и скелета.

Перед изменениями обязательно проверить:

- desktop;
- iPhone;
- iPad;
- загруженное видео;
- камера.

## 6. CSS

CSS переносить после JS-структуры.

Порядок:

1. токены;
2. кнопки;
3. карточки;
4. модальные окна;
5. шапка;
6. видео;
7. кабинеты;
8. мобильные стили.

## 7. Правило коммитов

Один коммит — одна понятная тема.

Хорошо:

- `Add project structure docs`
- `Extract local storage helpers`
- `Extract scoring helpers`
- `Refactor lesson card styles`

Плохо:

- `Fix everything`
- `Update app`
- `Design and logic changes`

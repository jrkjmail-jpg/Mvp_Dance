# Модульная система платформы

Документ фиксирует подход к архитектуре “Танцуй”: проект должен развиваться как модульная платформа, где новые возможности подключаются отдельными блоками и не ломают существующую логику.

## 1. Главная идея

Платформа должна состоять из ядра и подключаемых модулей.

Ядро отвечает за базовые сущности:

- пользователи;
- роли;
- студии;
- группы;
- уроки;
- ученики;
- доступы;
- хранение;
- безопасность;
- базовые настройки.

Модули отвечают за дополнительные возможности:

- автосдача;
- ручная проверка;
- достижения;
- рейтинги;
- видео-хранилище;
- родительские согласия;
- аналитика;
- уведомления;
- челленджи;
- тарифные функции.

## 2. Правило модульности

Любая новая возможность должна добавляться как отдельный модуль или как подмодуль существующего модуля.

Нельзя добавлять новую крупную функцию просто “в середину общего кода”.

Правильный подход:

```text
modules/
  auto-grading/
  achievements/
  manual-review/
  ratings/
```

Неправильный подход:

```text
app.js
  еще 500 строк новой логики
```

## 3. Внутренняя структура каждого модуля

Каждый модуль должен иметь одинаковую внутреннюю структуру.

Пример:

```text
modules/auto-grading/
  index.js
  config.js
  permissions.js
  data.js
  service.js
  ui/
    auto-grading-panel.js
    auto-grading-settings.js
  components/
    score-badge.js
    result-breakdown.js
  tests/
    auto-grading.test.js
  README.md
```

## 4. Назначение подблоков

### `index.js`

Точка входа в модуль. Через нее остальная система подключает модуль.

### `config.js`

Настройки модуля по умолчанию.

Например:

```js
export const autoGradingDefaults = {
  enabled: false,
  mode: "manual",
  allowRetake: true
};
```

### `permissions.js`

Проверка, кто может пользоваться модулем:

- директор студии;
- педагог;
- ученик;
- родитель;
- администратор платформы.

### `data.js`

Описание данных модуля:

- какие поля нужны в студии;
- какие поля нужны в уроке;
- какие поля нужны в результате;
- что хранится локально;
- что хранится на сервере.

### `service.js`

Основная бизнес-логика модуля.

Например:

- рассчитать оценку;
- сохранить результат;
- проверить настройки;
- подготовить данные для интерфейса.

### `ui/`

Экраны и крупные интерфейсные части модуля.

### `components/`

Маленькие переиспользуемые элементы:

- бейджи;
- кнопки;
- карточки;
- индикаторы;
- пустые состояния.

### `tests/`

Тесты модуля.

### `README.md`

Короткое описание модуля:

- зачем нужен;
- где используется;
- какие настройки имеет;
- какие данные хранит;
- от каких модулей зависит.

## 5. Уровни настроек

Модуль может иметь настройки на разных уровнях.

### Уровень платформы

Платформа решает, доступен ли модуль вообще.

```js
platform.modules = {
  autoGrading: true,
  achievements: true,
  ratings: false
};
```

### Уровень студии

Директор студии решает, включать ли функцию у себя.

```js
studio.modules = {
  autoGrading: true,
  manualReview: true,
  achievements: true,
  videoStorage: "skeleton-only"
};
```

### Уровень группы

Студия или педагог может задавать правила группы.

```js
group.settings = {
  autoGradingAllowed: true,
  parentConsentRequired: true,
  leaderboard: false
};
```

### Уровень урока

Педагог выбирает, как именно работает конкретный урок.

```js
lesson.settings = {
  gradingMode: "auto-and-manual",
  allowRetake: true,
  requireTeacherReview: false,
  showStars: true
};
```

### Уровень ученика

У ученика могут быть индивидуальные ограничения или доступы.

```js
student.settings = {
  canSubmitVideo: true,
  canUseCamera: true,
  visibleRatings: false
};
```

## 6. Пример: модуль автосдачи

Модуль автосдачи должен позволять выбирать режим проверки.

Возможные режимы:

- `manual` — только ручная проверка педагогом;
- `auto` — только автоматическая проверка;
- `auto-and-manual` — автоматическая проверка + подтверждение педагогом;
- `auto-as-hint` — автоматическая оценка только как подсказка педагогу.

Пример настроек урока:

```js
lesson.settings.gradingMode = "auto-and-manual";
```

Интерфейс педагога должен показывать это как понятный выбор:

- “Проверяет педагог”;
- “Проверяет алгоритм”;
- “Алгоритм + подтверждение педагога”;
- “Алгоритм только помогает педагогу”.

## 7. Пример: модуль достижений

Достижения должны быть отдельным модулем, а не встроенной частью карточки студии.

Уровни:

- достижения студии;
- достижения ученика;
- достижения урока;
- достижения группы.

Настройки:

```js
studio.modules.achievements = true;
lesson.settings.achievementEnabled = true;
student.achievements = [];
```

## 8. Пример: модуль видео-хранилища

Модуль должен поддерживать разные режимы хранения.

Варианты:

- `none` — видео не хранится;
- `skeleton-only` — хранится только скелет и результат;
- `last-attempt-video` — хранится только последняя попытка;
- `all-attempts-video` — хранятся все попытки;
- `teacher-only` — видео видит только педагог.

Пример:

```js
studio.modules.videoStorage = "skeleton-only";
```

## 9. Правило добавления нового модуля

Если появляется новая идея, сначала нужно ответить:

1. Это часть ядра или модуль?
2. Кто управляет этим модулем?
3. На каком уровне включается модуль?
4. Какие данные он хранит?
5. Какие права доступа нужны?
6. Где он отображается в интерфейсе?
7. Что будет, если модуль выключен?
8. Как протестировать модуль отдельно?

Только после этого добавлять код.

## 10. Модуль не должен ломать ядро

Каждый модуль должен быть отключаемым.

Если модуль выключен:

- приложение не должно падать;
- интерфейс не должен показывать лишние кнопки;
- данные модуля не должны требоваться для базовой работы;
- уроки, студии и ученики должны продолжать работать.

## 11. Будущая структура

Пример будущей структуры:

```text
src/
  core/
    users/
    studios/
    groups/
    lessons/
    access/
    billing/
    storage/

  modules/
    auto-grading/

  ui/
    components/
    layout/
    forms/

  app/
    init.js
    router.js
    state.js
```

## 12. Что важно для текущего MVP

В текущем MVP пока не нужно резко переносить весь код в `modules/`.

Сейчас задача:

- мыслить модульно;
- новые крупные функции описывать как модули;
- не добавлять большие куски логики в общий код без структуры;
- постепенно готовить проект к такому разделению.

Первый реальный шаг позже:

- выделить `storage`;
- потом `scoring`;
- потом `pose`;
- потом уже отдельные модули.

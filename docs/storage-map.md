# Storage map

Карта текущего клиентского хранения данных в MVP.

Документ нужен перед выносом `localStorage` в отдельный слой, а позже — перед миграцией на backend.

## 1. Общий принцип текущего MVP

Сейчас данные хранятся в браузере пользователя через `localStorage`.

Это удобно для прототипа, но не подходит как источник истины для реального продукта:

- данные не синхронизируются между устройствами;
- пользователь может очистить браузер и потерять данные;
- доступы нельзя защищать через `localStorage`;
- любые данные можно изменить вручную в браузере;
- нет серверного журнала действий.

## 2. Текущие ключи

### `danceReplayTheme`

Назначение:

- хранит выбранную тему интерфейса.

Формат:

```text
"light" | "dark"
```

Где используется:

- чтение темы при запуске;
- сохранение темы после переключения.

Будущее:

- можно оставить локально;
- не требует backend.

### `danceReplayAttempts`

Назначение:

- хранит последние попытки сдачи ученика.

Формат:

```js
[
  {
    score,
    stars,
    match,
    scores,
    seconds,
    lessonId,
    lessonName,
    lessonImage,
    at
  }
]
```

Где используется:

- история попыток;
- уровень и звезды ученика;
- блок пройденных уроков.

Будущее:

- должно уйти на backend;
- локально можно оставить только кэш.

### `danceReplayTeacherWorkspace`

Назначение:

- хранит студию, группы, подгруппы, уроки, учеников и активные выбранные элементы кабинета педагога.

Примерная структура:

```js
{
  studios: [],
  activeStudioId: "",
  activeGroupId: "",
  activeSubgroupIds: [],
  activeLessonId: ""
}
```

Где используется:

- кабинет педагога;
- страница студии;
- материалы студии;
- группы;
- уроки;
- ученики;
- приглашения.

Будущее:

- это один из первых кандидатов на перенос в backend;
- в будущем разбить на сущности: Studio, Group, Lesson, Student, Invite.

### `danceReplayStudentProfile`

Назначение:

- хранит профиль ученика.

Примерная структура:

```js
{
  id,
  firstName,
  lastName,
  nickname,
  birthDate,
  age,
  styles,
  coverImage,
  avatarImage,
  studioId,
  studioName,
  joinedAt
}
```

Где используется:

- регистрация ученика;
- карточка танцора;
- кабинет ученика;
- аватар в шапке;
- страница профиля.

Будущее:

- должно уйти на backend;
- локально можно хранить только кэш текущего пользователя.

### `danceReplayStudentSubmission`

Назначение:

- хранит последнюю опубликованную попытку ученика.

Где используется:

- публикация попытки;
- отображение результата.

Будущее:

- должно стать сущностью Submission / Attempt на backend.

### `danceReplayFavoriteLessons`

Назначение:

- хранит список избранных уроков.

Формат:

```js
["lesson-id-1", "lesson-id-2"]
```

Где используется:

- история / любимые уроки в кабинете ученика.

Будущее:

- можно синхронизировать через backend;
- локально оставить кэш.

### `danceReference:*`

Назначение:

- кэширует считанный эталон MediaPipe для видео педагога.

Ключ динамический:

```text
danceReference:{fileName}:{fileSize}:{duration}:{start}:{end}:{fps}
```

Формат:

```js
{
  savedAt,
  fps,
  start,
  end,
  frames
}
```

Где используется:

- загрузка ранее считанного эталона;
- ускорение повторного открытия урока.

Будущее:

- для MVP можно оставить локально;
- для платформы лучше хранить эталонные skeleton-данные на сервере после публикации урока.

## 3. Что можно оставить локально

Локально можно оставить:

- тему;
- временные UI-настройки;
- черновики;
- кэш эталона;
- кэш последнего открытого экрана.

## 4. Что нужно переносить на backend

На backend нужно переносить:

- профиль ученика;
- студии;
- группы;
- подгруппы;
- уроки;
- попытки;
- результаты;
- приглашения;
- согласия;
- доступы;
- журнал действий.

## 5. Безопасное направление рефакторинга

Первый технический шаг:

- создать единый storage-слой;
- заменить прямые вызовы `localStorage.getItem/setItem` на функции-обертки;
- не менять формат данных;
- не менять UI;
- не менять бизнес-логику.

Пример будущего API:

```js
readJson(key, fallback)
writeJson(key, value)
readTheme()
writeTheme(theme)
readTeacherWorkspace()
writeTeacherWorkspace(workspace)
readStudentProfile()
writeStudentProfile(profile)
readAttempts()
writeAttempts(attempts)
```

После этого внутреннюю реализацию будет проще заменить на backend API.

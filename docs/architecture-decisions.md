# Architecture decisions

Документ для фиксации важных решений. Если решение принято, его нужно записывать сюда, чтобы не возвращаться к одному и тому же спору в чате.

## ADR-001. MVP пока остается статическим frontend

Статус: принято временно.

Решение:

Текущий MVP продолжает работать как статическое приложение на GitHub + Cloudflare Pages. Backend не добавляется до согласования модели данных и ролей.

Причина:

Сначала нужно стабилизировать продуктовую структуру и не переносить хаос из frontend в backend.

## ADR-002. MediaPipe работает на клиенте

Статус: принято временно.

Решение:

MediaPipe остается на стороне устройства. Heavy/3D/smoothing не возвращаются без отдельного тестирования, потому что они ухудшали задержку скелета на мобильных устройствах.

Причина:

Для танцевального приложения задержка скелета хуже, чем менее сложная модель.

## ADR-003. Видео ученика не хранить по умолчанию

Статус: рабочая гипотеза.

Решение:

В будущем по умолчанию хранить результат, метаданные и точки скелета. Само видео ученика хранить только при отдельном согласии и необходимости.

Причина:

Видео детей — чувствительные данные. Лучше минимизировать хранение.

## ADR-004. Не делать большой родительский кабинет на первом этапе

Статус: рабочая гипотеза.

Решение:

На первом этапе сделать родительский шаг регистрации и согласий, но не полноценный отдельный кабинет родителя.

Причина:

Родитель важен юридически, но продуктовая ценность основного кабинета пока у ученика и студии.

## ADR-005. Рефакторинг делать слоями

Статус: принято.

Решение:

Сначала документация и структура. Потом storage, scoring, pose, media, UI, кабинеты.

Причина:

Проект уже хрупкий. Резкий перенос кода может сломать рабочий MVP.
